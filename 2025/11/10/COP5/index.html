计算指令

$$
\begin{align*}
&def:countOnes(x) = x 的前导一个数 \
&def:countZeros(x) = x 的前导零个数 \
&condition \leftarrow countZeros(GPR[rs]) > countOnes(GPR[rt]) \
&if \quad condition: \
&\qquad GPR[rd] \leftarrow {GPR[rs][31:16]，GPR[rt][15:0]} \
&else : \
&\qquad GPR[rd] \leftarrow {GPR[rt][31:16]，GPR[rs][15:0]}
\end{align*}
$$

比较基础，不需要修改顶层，过。

cjoc

$$
\begin{align*}
&ext \leftarrow signed_extend(imm||00) \
&temp \leftarrow ext[4:2] \
&data \leftarrow GPR[rs][temp+7,temp] \
&if\quad(data>8’d21): \
&\qquad GPR[rt] \leftarrow pc + 8\
&\qquad pc \leftarrow pc + 4 + ext \
&else : \
&\qquad 清空延迟槽指令()
\end{align*}
$$

寄在不会清空延迟槽，事实上只需要注意几点：

  1. cjoc暂停的时候不要清空FD，这个时候还没有产生跳转信号
  2. 当准备就绪的时候，由npc发出清空信号，而不是由FD寄存器发出信号（会产生毛刺）

load（不记得名字）

$$
\begin{align*}
&Addr \leftarrow GPR[base] + signed_extend(offset) \
&data \leftarrow Mem[Addr] \
&high_ones \leftarrow countOnes(data[31:16]) \
&low_ones \leftarrow countOnes(data[15:0]) \
&if\quad(high_ones > low_ones): \
&\qquad GPR[31] \leftarrow (data\quad xor\quad GPR[rt])\
&else : \
&\qquad GPR[high_ones] \leftarrow (data\quad xor\quad GPR[rt])
\end{align*}
$$

分析一下，只可能是 0~16 号寄存器，在该指令到 E 时阻塞需要用到这些寄存器的指令，在 M 阻塞 $T_{use}$ 为 0 且需要用到这些寄存器的指令，在 W 转发并写回

理论

多周期CPU

普林斯顿结构：指令和数据使用同一个内存

  • 将指令执行分解为多个步骤，每一步骤的执行，使用一个时钟周期
  • 在不同步骤执行中，可分时共享同一功能部件（如：Mem、ALU）
  • 每个步骤，增设寄存器，保存本步骤执行的结果，以供后续步骤使用

流水线

  • 可实现多个任务同时工作，但占用不同的资源
  • 流水线改善了整体工作负载的吞吐率，但不改善单个任务处理延迟(latency)
  • 流水线速率受限于最慢的流水段
  • 潜在加速比 = 流水线级数
  • 流水段执行时间不平衡，则加速比下降
  • 填充流水线和排放流水线，加速比下降

5 Stages of MIPS Datapath：

  1. IF: Instruction Fetch, Increment PC
  2. ID: Instruction Decode, Read Registers
  3. EX: Execution (ALU)
    Load/Store: Calculate Address
    Others: Perform Operation
  4. MEM:
    Load: Read Data from Memory
    Store: Write Data to Memory
  5. WB: Write Data Back to Register

流水线寄存器

命名法则：前级/后级

功能：时钟上升沿到来时，保存前级结果；之后输出至下级组合逻辑，也可能直接连接到下级流水线寄存器

后面的指令需要用到前面指令的结果产生的冒险

冻结：

行为：

  1. 冻结后续指令
  2. 清空当前指令所在流水线寄存器
  3. 冻结 PC

如何修改：增加流水线寄存器和 PC 使能信号，添加指令时间判断

取数指令延迟槽：即为插入的 nop

对编程的提示：

改善编译程序顺序（如在发生冒险的取数指令和计算指令之间放入一条后面需要算但是不产生冒险的指令），减少冒险，可以提高程序效率

3.控制冒险

例如分支指令需要使用计算结果决定是否跳转（这直接影响到下一条指令），但是计算结果需要经过 ALU 等获取（而这需要时间）

思路一：

先假设分支不发生，顺序执行，若后续需要跳转，清楚后续三条指令的所有相关流水线寄存器

优点：如果不需要跳转，就没有指令损失

缺点：清除逻辑比较复杂

思路二：

不使用 ALU ，而是在取数之后直接比较产生信号

优点：被浪费的指令可能减少到 1 条

缺点：如果依赖前序指令结果，则不能直接得到结果

改进方案：增加旁路（已经有结果） + 冻结（还没有结果）

分支延迟槽：可以发现两种思路都会运行后一条指令，所以可以插入 nop

跳转延迟槽：发生跳转时后续有一条指令已经取出，只有原地跳转才会使用，所以应该在跳转指令后插入 nop

jal：下一条指令应该是 PC + 8

命名规范

仅是个人习惯

划分为 F（Fetch）、D（Decode）、E（Execute）、M（Memory）、W（Writeback）区

流水线寄存器采用 X_Y_reg 命名，表示上游是 X 区，下游是 Y 区

采用分布式译码，X 区特有的控制器命名为 X_ctrl

实例化多选器时，命名为 dataChooser，表示选出 data 信号

关于数据线的命名：

X_data 表示的是在 X 区的 真实 数据，即已经经过冒险选择器的数据，而a_data 表示是从元件 a 发出的数据

7. producer & consumer

显然可以看出可以将指令集分类如下：

  1. cal_r：add、sub
  2. cal_i：ori、lui
  3. save：sw
  4. load：lw
  5. branch:beq
  6. jmode：j、jal
  7. jrmode：jr、jalr
type T_use data_requirement T_new(D/E) data_offer
cal_r 1 GPR[rs]、GPR[rt] 1 GPR[rd]
cal_i 1 GPR[rs] 1 GPR[rt]
load 1 GPR[rs] 2 GPR[rt]
save 1、2 GPR[rs] (1)、GPR[rt] (2) 0 -
branch 0 GPR[rs]、GPR[rt] 0 -
jmode 0 - 1 $0（j）、$ra(jal)
jrmode 0 GPR[rs] 1 GPR[rd]

说明：将 jr 后面当作 pc + 4 的加法指令，存入 rd ($0)

AT(address and time) 模型解释：

冒险发生的条件：$A_{use} == A_{new}$ （存在 $A_{new}$ 满足，若已经执行完毕排出，不会发生冒险，且 $0 不会影响）

使用暂停：$T_{use} < T_{new}$

使用转发：$T_{use} \geqslant T_{new}$

8.hazard_ctrl

应该算是本次实验最核心的部分，为了扩展性和封装性，我选择接收所有区域的指令到hazard_ctrl 内部计算 $T_{use}$ 和 $T_{new}$ 。

def name def name
input[31:0] F_D_instr output[2:0] D_rdata1Choose
input[31:0] D_E_instr output[2:0] D_rdata2Choose
input[31:0] E_M_instr output[2:0] E_rdata1Choose
input[31:0] M_W_instr output[2:0] E_rdata2Choose
output[2:0] M_rdata2Choose
output stall
output clear

选择信号（留的接口多就可以对齐）:

choose 000 001 010 011 100 101 110 111
D_rdata1Choose grf_rdata1 M_aluOut W_memOut W_aluOut - - - -
D_rdata2Choose grf_rdata2 M_aluOut W_memOut W_aluOut - - - -
E_rdata1Choose D_E_rdata1 M_aluOut W_memOut W_aluOut - - - -
E_rdata2Choose D_E_rdata2 M_aluOut W_memOut W_aluOut - - - -
M_rdata2Choose E_M_rdata2 - W_memOut W_aluOut - - - -

选择逻辑：

  1. 计算 F_D_instr 的 A（若不需要数据，则 A 值默认为 0） 和 $T_{use}$
  2. 计算后面三个寄存器中的 $T_{new}$
  3. 若产生冲突，根据最近的判断，如果 $T_{use} < T_{new}$，触发暂停，冻结PC、F_D_reg，排空 D_E_reg
  4. 若 $T_{new} = 0$ ，向前转发 A 值相同的

思考题

1、我们使用提前分支判断的方法尽早产生结果来减少因不确定而带来的开销，但实际上这种方法并非总能提高效率，请从流水线冒险的角度思考其原因并给出一个指令序列的例子。

原因是因为提前了寄存器数据的使用，如果数据正在被写但又没有产生不可转发，就必须暂停，浪费了时间

如果将 beq 放在 alu 中判断，就可以使用 M/W 寄存器转发的 ori 结果；
如果提前分支判断，由于读出寄存器数据和ALU产生结果在同一周期，无法转发

2、因为延迟槽的存在，对于 jal 等需要将指令地址写入寄存器的指令，要写回 PC + 8，请思考为什么这样设计？

jal 指令后默认是延迟槽指令，而由于延迟槽的设计该指令已经被运行过，实际上 jal 希望链接顺序的下一条未被执行的指令，因此要写入 PC + 8

3、我们要求大家所有转发数据都来源于流水寄存器而不能是功能部件（如 DM 、 ALU ），请思考为什么？

在基本的设计中各区域已经基本平衡，从流水寄存器转发相当于开启一个新的阶段，不会增加区域关键路径；如果直接从功能部件转发，相当于把两个流水段连在一起，大大增加了该流水段的关键路径，为了照顾这个设计必须降低时钟频率，得不偿失

4、我们为什么要使用 GPR 内部转发？该如何实现？

因为在 D（解码）阶段和 W（写回）阶段都是对 GPR 进行操作，如果需要读的寄存器与要写的重合，此时在上升沿之前 GPR 里的数据还没更新，如果不进行内部转发就会导致读出来的数据是修改前的，与实际指令不符合

在读出数据后加多选器，另一路连要写回的数据，由冒险控制器判断是否产生对应冒险并选出数据

5、我们转发时数据的需求者和供给者可能来源于哪些位置？共有哪些转发数据通路？

就所给数据集而言，设计转发所需的数据都是已经产生但还未写入寄存器的数据

因此供给者可以来自 E/M_reg(ALU) 、 M/W_reg(DM、ALU)

需求者则是 nPC(GPR[rs]、GPR[rt]) 、 ALU(GPR[rs]、GPR[rt]) 、 DM(GPR[rt])

7、确定你的译码方式，简要描述你的译码器架构，并思考该架构的优势以及不足。

我采用分布式译码，每个流水段配置相应的controller和splitter，并流水指令instr，在每个流水段都进行译码产生控制信号。

优势是能够保留足够多的信息，便于后面使用计算

缺点是占用较多的存储资源，也会略微增加延迟
